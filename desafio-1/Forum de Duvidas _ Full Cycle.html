const url = "https://economia.awesomeapi.com.br/json/last/USD-BRL"

var db *sql.DB

const requestTimeout = 200 * time.Millisecond
const dbTimeout = 10 * time.Millisecond

type Cotacao struct {
  Usdbrl struct {
    Code       string `json:"code"`
    Codein     string `json:"codein"`
    Name       string `json:"name"`
    High       string `json:"high"`
    Low        string `json:"low"`
    VarBid     string `json:"varBid"`
    PctChange  string `json:"pctChange"`
    Bid        string `json:"bid"`
    Ask        string `json:"ask"`
    Timestamp  string `json:"timestamp"`
    CreateDate string `json:"create_date"`
  } `json:"USDBRL"`

func initializeDB() (*sql.DB, error) {
  db, err := sql.Open("sqlite3", "cotacoes.db")
  if err != nil {
    return nil, err
  }

  query := `
    CREATE TABLE IF NOT EXISTS cotacoes (
        code TEXT,
    codein TEXT,
    name TEXT,
    high TEXT,
    low TEXT,
    varBid TEXT,
    pctChange TEXT,
    bid TEXT,
    ask TEXT,
    timestamp TEXT,
    create_date TEXT
    );`
  _, err = db.Exec(query)
  if err != nil {
    return nil, err
  }

  return db, nil
}

func logErrorConsoleAndBrowser(msg string, w http.ResponseWriter) {
  log.Println(msg)
  http.Error(w, msg, http.StatusInternalServerError)
}

func getCotacao(ctxBg context.Context, data *Cotacao) (*Cotacao, error) {
  ctx, cancel := context.WithTimeout(ctxBg, requestTimeout)
  defer cancel()

  req, err := http.NewRequestWithContext(ctx, "GET", url, nil)
  if err != nil {
    errPersonalizado := errors.New("Erro ao preparar a requisio: \n" + err.Error())
    return &Cotacao{}, errPersonalizado
  }
  res, err := http.DefaultClient.Do(req)
  if err != nil {
    errPersonalizado := errors.New("Erro ao fazer a requisio: \n" + err.Error())
    return &Cotacao{}, errPersonalizado
  }

  body, err := io.ReadAll(res.Body)

    return &Cotacao{}, errPersonalizado

    body,
    data,
  )

  if err != nil {
    errPersonalizado := errors.New("Erro json.Unmarshal: \n" + err.Error() + "\n" + err.Error())
    return &Cotacao{}, errPersonalizado
  }

Melhorias na Segurana do Cdigo:

Para melhorar a segurana do cdigo que foi submetido, uma das principais medidas prevenir ataques de injeo de SQL. Uma forma de fazer isso utilizando prepared statements ao invs de concatenar diretamente as strings das consultas SQL. Isso ajuda a separar os dados dos comandos SQL, evitando a execuo de comandos maliciosos.

Exemplo de Cdigo com Prepared Statement:

// Funo para inserir dados no banco de dados utilizando prepared statement
func insertWithTimeout(ctx context.Context, db *sql.DB, data *Cotacao) error {
    query := "INSERT INTO cotacoes (moeda, valor) VALUES (?, ?)"

    stmt, err := db.PrepareContext(ctx, query)
    if err != nil {
        return err
    }
    defer stmt.Close()

    _, err = stmt.ExecContext(ctx, data.Moeda, data.Valor)
    if err != nil {
        return err
    }

    return nil
}

Neste exemplo, a consulta SQL preparada com placeholders ? para os valores que sero inseridos. Em seguida, o mtodo PrepareContext utilizado para criar um statement preparado. Ao chamar ExecContext com os valores apropriados, o banco de dados ir tratar os dados de forma segura, evitando a injeo de SQL.

Explicao do %v em log.Printf("Erro: %s - %v", msg, err):

  • O %v em log.Printf um verb que utilizado para formatar e exibir o valor de uma varivel de forma genrica. Ele utilizado para imprimir o valor da varivel err no exemplo fornecido.

  • Quando %v utilizado em conjunto com log.Printf, ele formata o valor da varivel de acordo com o seu tipo. Por exemplo, se err for uma string, %v ir exibir a string. Se err for um erro, %v ir exibir a mensagem de erro.

  • Dessa forma, %v uma forma genrica de exibir o valor de uma varivel em um log, independentemente do seu tipo. Isso til para exibir informaes sobre variveis de diferentes tipos em mensagens de log.

Por que a segurana vai melhorar usando PrepareContext sendo que ExecContext j utiliza placeholders?
IA Full Cycle Moderador Beta 8 minutos atrs

O uso de PrepareContext em conjunto com ExecContext em operaes de banco de dados pode melhorar a segurana do cdigo, mesmo que ExecContext j utilize placeholders para os valores das consultas. Vamos entender melhor o motivo:

Razes para Utilizar PrepareContext:

  1. Preveno de Injeo de SQL:

    • O uso de PrepareContext cria um statement preparado que enviado ao banco de dados. Isso permite que o banco de dados compile a consulta SQL uma vez e a otimize para execues subsequentes. Alm disso, o banco de dados pode realizar a validao dos placeholders e dos tipos de dados, ajudando a prevenir ataques de injeo de SQL.

  2. Reutilizao de Consultas:

    • Com PrepareContext, possvel reutilizar o statement preparado para executar consultas com diferentes valores. Isso pode melhorar o desempenho em operaes repetidas, pois a consulta SQL j est compilada e otimizada.

  3. Clareza e Manutenibilidade do Cdigo:

    • O uso de PrepareContext separa a preparao da consulta SQL da execuo da mesma, tornando o cdigo mais claro e modular. Isso facilita a manuteno e compreenso do cdigo.

Complemento ao ExecContext:
