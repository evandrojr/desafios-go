O client.go deverá realizar uma requisição HTTP no server.go solicitando a cotação do dólar.

O server.go deverá consumir a API contendo o câmbio de Dólar e Real no endereço: https://economia.awesomeapi.com.br/json/last/USD-BRL e em seguida deverá retornar no formato JSON o resultado para o cliente.

Usando o package "context", o server.go deverá registrar no banco de dados SQLite cada cotação recebida, sendo que o timeout máximo para chamar a API de cotação do dólar deverá ser de 200ms e o timeout máximo para conseguir persistir os dados no banco deverá ser de 10ms.

O client.go precisará receber do server.go apenas o valor atual do câmbio (campo "bid" do JSON). Utilizando o package "context", o client.go terá um timeout máximo de 300ms para receber o resultado do server.go.

Os 3 contextos deverão retornar erro nos logs caso o tempo de execução seja insuficiente.

O client.go terá que salvar a cotação atual em um arquivo "cotacao.txt" no formato: Dólar: {valor}

O endpoint necessário gerado pelo server.go para este desafio será: /cotacao e a porta a ser utilizada pelo servidor HTTP será a 8080.

const url = "https://economia.awesomeapi.com.br/json/last/USD-BRL"

var db *sql.DB

const requestTimeout = 200 * time.Millisecond
const dbTimeout = 10 * time.Millisecond

type Cotacao struct {
  Usdbrl struct {
    Code       string `json:"code"`
    Codein     string `json:"codein"`
    Name       string `json:"name"`
    High       string `json:"high"`
    Low        string `json:"low"`
    VarBid     string `json:"varBid"`
    PctChange  string `json:"pctChange"`
    Bid        string `json:"bid"`
    Ask        string `json:"ask"`
    Timestamp  string `json:"timestamp"`
    CreateDate string `json:"create_date"`
  } `json:"USDBRL"`
}

func initializeDB() (*sql.DB, error) {
  db, err := sql.Open("sqlite3", "cotacoes.db")
  if err != nil {
    return nil, err
  }

  query := `
    CREATE TABLE IF NOT EXISTS cotacoes (
        code TEXT,
    codein TEXT,
    name TEXT,
    high TEXT,
    low TEXT,
    varBid TEXT,
    pctChange TEXT,
    bid TEXT,
    ask TEXT,
    timestamp TEXT,
    create_date TEXT
    );`
  _, err = db.Exec(query)
  if err != nil {
    return nil, err
  }

  return db, nil
}

func logErrorConsoleAndBrowser(msg string, w http.ResponseWriter) {
  log.Println(msg)
  http.Error(w, msg, http.StatusInternalServerError)
}

func getCotacao(ctxBg context.Context, data *Cotacao) (*Cotacao, error) {
  ctx, cancel := context.WithTimeout(ctxBg, requestTimeout)
  defer cancel()

  req, err := http.NewRequestWithContext(ctx, "GET", url, nil)
  if err != nil {
    errPersonalizado := errors.New("Erro ao preparar a requisição: \n" + err.Error())
    return &Cotacao{}, errPersonalizado
  }
  res, err := http.DefaultClient.Do(req)
  if err != nil {
    errPersonalizado := errors.New("Erro ao fazer a requisição: \n" + err.Error())
    return &Cotacao{}, errPersonalizado
  }

  defer res.Body.Close()

  body, err := io.ReadAll(res.Body)
  if err != nil {
    errPersonalizado := errors.New("Erro ao ler a requisição: \n" + err.Error())
    return &Cotacao{}, errPersonalizado
  }

  err = json.Unmarshal(
    body,
    data,
  )

  if err != nil {
    errPersonalizado := errors.New("Erro json.Unmarshal: \n" + err.Error() + "\n" + err.Error())
    return &Cotacao{}, errPersonalizado
  }

Melhorias na Segurança do Código:

Para melhorar a segurança do código que foi submetido, uma das principais medidas é prevenir ataques de injeção de SQL. Uma forma de fazer isso é utilizando prepared statements ao invés de concatenar diretamente as strings das consultas SQL. Isso ajuda a separar os dados dos comandos SQL, evitando a execução de comandos maliciosos.

Exemplo de Código com Prepared Statement:

// Função para inserir dados no banco de dados utilizando prepared statement
func insertWithTimeout(ctx context.Context, db *sql.DB, data *Cotacao) error {
    query := "INSERT INTO cotacoes (moeda, valor) VALUES (?, ?)"

    stmt, err := db.PrepareContext(ctx, query)
    if err != nil {
        return err
    }
    defer stmt.Close()

    _, err = stmt.ExecContext(ctx, data.Moeda, data.Valor)
    if err != nil {
        return err
    }

    return nil
}

Neste exemplo, a consulta SQL é preparada com placeholders ? para os valores que serão inseridos. Em seguida, o método PrepareContext é utilizado para criar um statement preparado. Ao chamar ExecContext com os valores apropriados, o banco de dados irá tratar os dados de forma segura, evitando a injeção de SQL.

Explicação do %v em log.Printf("Erro: %s - %v", msg, err):

  • O %v em log.Printf é um verb que é utilizado para formatar e exibir o valor de uma variável de forma genérica. Ele é utilizado para imprimir o valor da variável err no exemplo fornecido.

  • Quando %v é utilizado em conjunto com log.Printf, ele formata o valor da variável de acordo com o seu tipo. Por exemplo, se err for uma string, %v irá exibir a string. Se err for um erro, %v irá exibir a mensagem de erro.

  • Dessa forma, %v é uma forma genérica de exibir o valor de uma variável em um log, independentemente do seu tipo. Isso é útil para exibir informações sobre variáveis de diferentes tipos em mensagens de log.

Por que a segurança vai melhorar usando PrepareContext sendo que ExecContext já utiliza placeholders?
IA Full Cycle Moderador Beta 8 minutos atrás

O uso de PrepareContext em conjunto com ExecContext em operações de banco de dados pode melhorar a segurança do código, mesmo que ExecContext já utilize placeholders para os valores das consultas. Vamos entender melhor o motivo:

Razões para Utilizar PrepareContext:

  1. Prevenção de Injeção de SQL:

    • O uso de PrepareContext cria um statement preparado que é enviado ao banco de dados. Isso permite que o banco de dados compile a consulta SQL uma vez e a otimize para execuções subsequentes. Além disso, o banco de dados pode realizar a validação dos placeholders e dos tipos de dados, ajudando a prevenir ataques de injeção de SQL.

  2. Reutilização de Consultas:
